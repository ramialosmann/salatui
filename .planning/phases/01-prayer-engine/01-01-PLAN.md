<task type="auto">
  <name>Task 1: Initialize Cargo project and create config module with tests</name>
  <files>Cargo.toml, src/config.rs</files>
  <action>
Initialize Rust binary crate with `cargo init` (the project directory already exists but has no Cargo.toml).

Add all dependencies to Cargo.toml:
```toml
[dependencies]
salah = "0.7"
clap = { version = "4.5", features = ["derive"] }
serde = { version = "1.0", features = ["derive"] }
toml = "1.0"
dirs = "6.0"
chrono = { version = "0.4", features = ["serde"] }
anyhow = "1"
```

Create `src/config.rs` with:

1. Config structs with serde Deserialize + Serialize derives:
   - `AppConfig` with fields: `location: LocationConfig`, `calculation: CalculationConfig`, `display: DisplayConfig` -- all with `#[serde(default)]`
   - `LocationConfig` (Default impl): `lat: Option<f64>`, `lon: Option<f64>`
   - `CalculationConfig` with defaults: `method: String` (default "mwl"), `madhab: String` (default "shafi")
   - `DisplayConfig` with defaults: `time_format: String` (default "24h")

2. `config_path() -> anyhow::Result<PathBuf>`: Uses `dirs::config_dir()` to return `~/.config/tui-adhan/config.toml`. Error if config dir cannot be determined.

3. `generate_default_config(path: &Path) -> anyhow::Result<()>`: Creates parent dirs with `create_dir_all`, writes a raw string literal containing commented TOML (NOT serialized -- to preserve helpful comments). The template should list all supported methods and madhab options in comments. Lat/lon should be commented out to force user to set them.

4. `load_or_create_config() -> anyhow::Result<AppConfig>`: If config file exists, read and parse. If not, generate default and parse it (which will have None lat/lon). Handle file permission errors gracefully.

5. `merge_config_with_cli(config: AppConfig, cli: &Cli) -> AppConfig`: Overlay CLI Option fields onto config. If CLI field is Some, use it; otherwise keep config value. For lat/lon, CLI Some overrides config. For method/madhab, CLI Some overrides config string.

6. Unit tests module (`#[cfg(test)] mod tests`):
   - `test_parse_config`: Parse a valid TOML string with all fields set, assert values match
   - `test_parse_config_defaults`: Parse empty string, assert defaults (method="mwl", madhab="shafi", time_format="24h", lat=None, lon=None)
   - `test_cli_overrides_config`: Create config with method="mwl", create CLI with method=Some("egyptian"), merge and assert method is "egyptian"
   - `test_madhab_config`: Parse TOML with madhab="hanafi", assert it deserializes correctly

Note: The `merge_config_with_cli` function takes a `Cli` struct reference -- this creates a dependency on cli.rs, but both are created in this plan. Import `Cli` from `crate::cli`. Write the config module first, then cli.rs in Task 2, then make sure both compile together.

<task type="auto">
  <name>Task 2: Create CLI module and wire main.rs with config validation</name>

  <action>
Create `src/cli.rs` with clap derive:
```rust
use clap::Parser;

#[derive(Parser, Debug)]
#[command(name = "tui-adhan", about = "Islamic prayer times in your terminal")]
pub struct Cli {
    /// Latitude (overrides config)
    #[arg(long)]
    pub lat: Option<f64>,
    /// Longitude (overrides config)
    #[arg(long)]
    pub lon: Option<f64>,
    /// Calculation method (overrides config). Options: mwl, egyptian, karachi, umm_al_qura, dubai, moonsighting_committee, north_america, isna, kuwait, qatar, singapore, tehran, turkey
    #[arg(long)]
    pub method: Option<String>,
    /// Asr madhab: shafi or hanafi (overrides config)
    #[arg(long)]
    pub madhab: Option<String>,
}
```

Wire `src/main.rs`:
1. Declare modules: `mod cli; mod config;`
2. In `main() -> anyhow::Result<()>`:
   - Parse CLI with `Cli::parse()`
   - Load config with `load_or_create_config()?`
   - Merge with `merge_config_with_cli(config, &cli)`
   - Validate lat/lon: if either is None, print helpful error to stderr pointing to config file path and CLI usage, then `std::process::exit(1)`
   - For now, if lat/lon are valid, print a placeholder message with the coordinates (prayer calculation comes in Plan 02)
3. Add `mod prayer;` declaration as a placeholder (create empty `src/prayer.rs` file so it compiles)

The main function should follow the exact error handling pattern from RESEARCH.md: exit code 1 for missing lat/lon, let anyhow handle other errors.
